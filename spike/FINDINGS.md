## 5. Planning resolution

**Amendments to M2 scope:**
- Station/"at stop" derivation is mandatory: geofence against `stops.txt` +
  cross-reference Trip Updates' predicted arrival/departure times. Not
  optional, and not a fallback path — VP never gives this directly.
- Poll interval stays at 10s (already the floor; confirmed correct against
  measured cadence, no change needed).
- trip_id reconciliation: direct static join only. No fuzzy matching or
  Trip-Updates-based reconciliation logic needed — drop it from scope,
  simplifies M2.
- Staleness alerting keys off the feed header timestamp going stale, not
  off entity count — a 0-entity poll with an advancing header is normal
  overnight service and must never alert.
- New: use the `x-rate-limit` response header (JSON, multiple throttle
  windows) for adaptive backoff instead of a hardcoded quota assumption —
  wasn't in original scope, cheap to add.
- Conditional GET is not supported by the API (no ETag/Last-Modified) —
  plain GET + header-timestamp dedupe, as already planned. Confirms this
  wasn't a missed optimization.

**Amendments to M4 scope:**
- Ghost-train fallback (render scheduled position from static timetable
  when a trip is missing from the live feed) is now required, not
  optional — coverage gaps are real (as low as 33-67% in some bands) and
  this fallback is load-bearing, not a nicety. Needs proper design
  attention, not an afterthought.
- City Loop handling: freeze-in-place as the baseline (most freezes are
  short, ~20s, consistent with a normal station dwell), plus two
  guardrails: a max-freeze timeout (past which the train shows as
  "position uncertain" rather than a literally frozen dot) and
  teleport-distance smoothing for the rare large jumps (up to 863m) rather
  than an instant snap. Full dead-reckoning is likely unnecessary given how
  rare/short most freezes are.

**New risks surfaced by the data:**
- Coverage gaps are real and uneven across time-of-day bands (not just an
  overnight artifact) — the ghost-train fallback needs to handle all bands,
  not just quiet periods.
- The static GTFS portal regenerates trip_ids on every publish (44% churn
  in the full trip_id set between two pulls 3 days apart) — trip_ids for
  already-elapsed/current dates stayed stable in this spike, but a
  long-running production service must not assume future trip_ids survive
  a republish. M2 needs a static-feed refresh job; monitor the portal's
  actual update cadence rather than assuming.
- A 1,614s (~27min) VP gap coincided with a logged read-timeout error —
  the production poller's backoff/retry needs to survive multi-minute
  outages without cascading.

**Assumptions invalidated:** None outright — Q1's finding (station state
must be derived, not read directly) was already anticipated from the
2026-07-17 smoke test and is now confirmed at full scale, not overturned.

**Deferred items resolved / re-deferred:**

- Ghost-train fallback — **resolved**: build it, required for M4.
- City Loop dead-reckoning approach — **resolved**: freeze-in-place +
  guardrails, not full dead-reckoning.

**Verdict:** Loop-gap share is ~0%, not ~99% — the revisit clause does
**not** trigger. Consistent with Q4's original finding: the loop tunnel

loop. **Resolution:** keep the generic ghost state machine as designed in
CLAUDE.md — no dedicated "in loop, position unavailable" state needed.
